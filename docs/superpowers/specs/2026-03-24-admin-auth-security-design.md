# 课堂签到系统后台独立登录与安全收口设计

- 日期：2026-03-24
- 状态：已完成设计讨论，待用户审阅
- 适用项目：`/Volumes/HIKSEMI/Python/签到系统`

## 1. 背景

当前项目是单教师课堂签到系统，学生通过设备绑定和 4 位签到码完成签到，教师通过后台创建场次、查看实时页、导入学生和查看统计。

现状中存在几类明显风险：

- 管理后台入口与学生签到入口耦合，首页会基于设备 UUID 自动跳转后台。
- 管理员权限部分依赖前端可写的 `device_id` cookie，信任边界不清晰。
- `ADMIN_PASSWORD`、`ADMIN_DEVICE_UUIDS`、`ADMIN_ENTRY_*` 三套后台入口配置并存，逻辑分散且容易漂移。
- Supabase 当前关闭了 RLS，并向 `anon/authenticated` 暴露了直接写权限。
- 服务端 Supabase 客户端允许在缺少服务端密钥时回退到 publishable key，导致权限模型混乱。

本设计的目标是把“学生签到系统”和“教师后台系统”的边界切清楚，并为后续实现提供可执行的改造方案。

## 2. 目标

- 将教师后台改为独立登录入口，不再与学生签到入口共享进入路径。
- 管理员权限只来源于服务端签发的后台会话，不再依赖设备标识。
- 保留环境变量密码方案，避免为当前项目引入重型身份系统。
- 收紧 Supabase 权限，让浏览器无法直接写业务表。
- 保持学生绑定、查看签到上下文、输入签到码三条主流程继续可用。

## 3. 非目标

- 本次不引入管理员账号体系、多用户后台或第三方登录。
- 本次不把学生侧升级为账号登录系统。
- 本次不设计复杂的验证码、短信验证或外部 IAM。
- 本次不重构与安全目标无关的页面视觉或后台信息架构。

## 4. 设计决策概览

### 4.1 管理后台改为独立登录流

- 学生入口只保留 `/`、`/bind`、`/sign`。
- 新增固定后台登录页 `/admin/login`。
- `/admin` 只承担跳转职责：
  - 已登录时跳到 `/admin/session`
  - 未登录时跳到 `/admin/login`
- 不再允许首页根据教师设备 UUID 自动进入后台。

### 4.2 管理员权限只来自后台会话

- 管理员通过环境变量 `ADMIN_PASSWORD` 完成登录。
- 登录成功后，由服务端签发 `httpOnly` 的管理员会话 cookie。
- 后台受保护页面和后台 API 只校验管理员会话，不校验 `device_id`。
- 学生设备标识与管理员授权彻底解耦。

### 4.3 移除旧的后台放行机制

以下机制全部删除：

- `ADMIN_DEVICE_UUIDS`
- `ADMIN_ENTRY_USERNAME`
- `ADMIN_ENTRY_PASSWORD`
- 后台入口白名单设备自动放行
- 单独的 admin entry 路由与相关 cookie

这样做的结果是：后台只有一条清晰的认证路径，环境配置和代码逻辑保持一致。

### 4.4 服务端必须使用服务端密钥访问 Supabase

- `getSupabaseAdmin()` 必须要求 `SUPABASE_SECRET_KEY` 或 `SUPABASE_SERVICE_ROLE_KEY` 存在。
- 不再允许回退到 `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_DEFAULT_KEY`。
- 缺少服务端密钥时直接报错并阻止启动相关数据操作。

## 5. 信任边界

### 5.1 浏览器侧

浏览器只负责：

- 采集学生输入
- 读取和持有学生设备标识
- 请求 Next.js 提供的同源 API
- 展示绑定结果、签到上下文和签到结果

浏览器不再拥有以下能力：

- 直接写 Supabase 业务表
- 基于设备标识获得后台管理权限
- 通过篡改 URL 参数获得更高信任级别

### 5.2 Next.js 服务端

服务端负责：

- 验证管理员登录密码
- 生成和校验管理员会话
- 校验学生绑定与签到请求
- 使用服务端密钥访问 Supabase
- 汇总数据库错误并返回对用户可理解的消息

### 5.3 Supabase

Supabase 只接受来自服务端受控路径的业务写入。数据库层通过 RLS 和授权策略兜底，即使前端请求被篡改，也不能直接写入核心表。

## 6. 路由与认证方案

### 6.1 路由调整

- `/`：学生首页，不再承担后台入口功能。
- `/bind`：学生绑定页。
- `/sign`：学生签到页。
- `/admin/login`：新的教师后台登录页。
- `/admin`：后台入口跳转页。
- `/admin/session`、`/admin/session/[id]`、`/admin/stats`、`/admin/students`：后台受保护页面。

### 6.2 页面行为

- 学生首页仅显示学生相关状态，不再因为设备标识触发后台跳转。
- 访问后台受保护页面时，若管理员未登录，统一重定向到 `/admin/login`。
- 管理员登录成功后进入 `/admin/session`。
- 管理员退出后只清后台会话，不影响学生设备绑定状态。

### 6.3 后台会话

管理员会话继续采用签名 token + `httpOnly` cookie 的轻量方式：

- cookie 为 `httpOnly`
- `sameSite=lax`
- 生产环境启用 `secure`
- 限制过期时间
- 所有后台页面与后台 API 统一使用该会话校验

该方案符合当前单教师系统体量，不额外引入身份服务。

## 7. 学生设备标识与签到防伪

### 7.1 设备标识定位

`device_id` 的唯一用途是标识一台学生设备，用于：

- 首次绑定设备到学生
- 查询当前设备是否已绑定学生
- 在签到时定位该设备对应的学生

`device_id` 不是权限凭证，不得用于授予后台管理权限。

### 7.2 读取策略

服务端接口对学生设备标识采用以下策略：

- 优先从同源 cookie 读取 `device_id`
- 不再以 URL 查询参数作为可信主来源
- 如需保留参数兼容，也只作为低优先级兜底，并逐步移除

这样可以降低设备标识出现在地址栏、日志、截图和分享链接中的概率。

### 7.3 签到校验链

学生签到成功必须同时满足：

- 请求来自已绑定设备
- 设备绑定的学生存在
- 学生班级与当前签到场次班级匹配
- 当前场次存在且状态有效
- 4 位签到码正确
- 该学生在该场次尚未签到

数据库仍保留唯一索引 `attendance_records(session_ref, student_ref)` 作为最终兜底，避免并发下重复签到。

## 8. Supabase 权限方案

### 8.1 总体原则

当前项目不需要浏览器直连 Supabase 进行业务写入，因此采用“浏览器不写库，只有服务端写库”的策略。

### 8.2 权限收口

需要进行以下调整：

- 为 `students`、`device_bindings`、`attendance_sessions`、`attendance_records` 全部开启 RLS。
- 去掉面对 `anon` / `authenticated` 的直接写权限。
- 默认不允许前端身份直接修改业务表。
- 所有新增、更新、删除动作统一收口到 Next.js API。

### 8.3 未来扩展

如果后续确实需要浏览器直连 Supabase 的只读场景，应单独新增最小只读 policy，而不是恢复通用写权限。

## 9. 清理范围

### 9.1 需要删除的配置与逻辑

- `ADMIN_DEVICE_UUIDS`
- `ADMIN_ENTRY_USERNAME`
- `ADMIN_ENTRY_PASSWORD`
- admin entry cookie
- 设备白名单自动进入后台逻辑
- 独立 admin entry 提交路由

### 9.2 需要保留的配置

- `ADMIN_PASSWORD`
- `SUPABASE_SECRET_KEY` 或 `SUPABASE_SERVICE_ROLE_KEY`
- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_DEFAULT_KEY`

说明：

- `NEXT_PUBLIC_*` 变量仍可用于前端初始化公开客户端，但不再承担服务端特权用途。
- 若当前实现实际上未使用浏览器 Supabase 客户端，可在后续实现阶段继续评估是否保留公开 key。

## 10. 迁移步骤

建议按以下顺序实施：

1. 调整后台入口与登录页路由，建立 `/admin/login` 独立登录流。
2. 将后台页面和后台 API 的权限判断统一为管理员会话。
3. 删除 `ADMIN_DEVICE_UUIDS` 和 `ADMIN_ENTRY_*` 相关代码、配置和 README 描述。
4. 将学生设备相关接口改为优先使用 cookie 读取 `device_id`。
5. 收紧 `getSupabaseAdmin()` 的密钥要求，取消 publishable key fallback。
6. 更新 `schema.sql`，开启 RLS 并移除前端直接写权限。
7. 回归验证学生绑定、签到和后台操作主流程。

这个顺序的目的，是先稳定应用层的认证与路由，再切数据库权限边界，降低联调复杂度。

## 11. 错误处理原则

- 管理员未登录访问后台：统一重定向到 `/admin/login`。
- 管理员密码错误：停留在登录流并显示明确错误。
- 缺少服务端 Supabase 密钥：服务端直接报配置错误，不静默回退。
- 学生未绑定设备：返回明确提示，引导进入绑定页。
- 签到码错误、场次关闭、班级不匹配、重复签到：保留现有明确业务错误语义。

## 12. 测试策略

### 12.1 后台认证

- 未登录访问后台页面应跳转到 `/admin/login`
- 未登录访问后台 API 应被拒绝或重定向
- 错误密码不能登录
- 正确密码登录后可访问后台
- 刷新页面后仍保持后台登录状态
- 退出登录后立即失去后台访问权限

### 12.2 学生主流程

- 首页不再自动进入后台
- 学生可正常绑定设备
- 已绑定设备可正常获取签到上下文
- 正确签到码可完成签到
- 重复签到只返回“已完成签到”

### 12.3 权限与边界

- 伪造 `device_id` 不应获得后台权限
- 浏览器无法直接向 Supabase 业务表写入数据
- 服务端缺少 Supabase 服务端密钥时应明确失败

## 13. 风险与权衡

### 13.1 接受的权衡

- 后台仍采用单密码模式，安全性低于正式账号体系，但与当前项目规模匹配。
- 学生侧仍使用设备标识而不是账号登录，存在设备共享或本地篡改的天然上限，但对课堂签到场景是可接受的。

### 13.2 规避的风险

本设计重点规避以下高风险问题：

- 学生设备标识越权进入后台
- 前端直接写数据库
- 服务端与前端权限边界混用
- 旧入口配置残留导致行为不可预测

## 14. 实施完成后的预期状态

实施完成后，系统应呈现以下状态：

- 学生与教师各走各的入口，路径清晰。
- 后台只有一条认证路径：`/admin/login` + `ADMIN_PASSWORD`。
- `device_id` 只代表学生设备，不代表管理员身份。
- 所有业务写操作都经过 Next.js 服务端。
- Supabase 成为真正的后端数据层，而不是前端可直接操作的开放数据库。
