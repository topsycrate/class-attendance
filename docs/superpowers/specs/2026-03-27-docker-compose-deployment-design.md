# 课堂签到系统 Docker Compose 部署设计

- 日期：2026-03-27
- 状态：已完成方案选择，待用户审阅
- 适用项目：`/Volumes/HIKSEMI/Python/签到系统`

## 1. 背景

当前项目是一个基于 Next.js 15 的单体 Web 应用，业务数据已经托管在外部 Supabase。用户已经决定采用“服务器运行 Docker Compose 应用容器，反向代理继续使用服务器现有组件”的部署方式。

这意味着本次工作重点不是重建一整套基础设施，而是让现有项目具备以下能力：

- 可在服务器上稳定构建与启动
- 运行环境与本地开发解耦
- 环境变量、镜像内容和启动命令可重复
- 后续更新时只需要重新构建并重启容器

## 2. 目标

- 为当前 Next.js 项目提供可直接用于服务器部署的 Docker 化方案。
- 采用 `docker compose` 管理单个应用服务，数据库仍使用外部 Supabase。
- 保持现有路由与业务逻辑不变，不引入新的中间件或基础设施依赖。
- 补齐部署文档，使服务器部署步骤可复制。

## 3. 非目标

- 本次不把 Supabase 本地化或容器化。
- 本次不在 Compose 中额外引入 Nginx、Caddy、Redis、Postgres 等服务。
- 本次不改造为 Kubernetes、Swarm 或 CI/CD 自动发布流水线。
- 本次不新增与部署无关的业务功能。

## 4. 方案对比与结论

### 4.1 备选方案

本次讨论过三种部署路径：

- 直接在服务器上用 `pm2` 或 `systemd` 裸跑 `next start`
- 用 `docker compose` 运行 Next.js 容器，并复用服务器现有反向代理
- 用 `docker compose` 同时管理 Next.js 与反向代理

### 4.2 选择结果

选择第二种方案：

- Compose 只管理应用容器
- 现有 Nginx / Caddy / 宝塔继续承担 80/443 入口和 TLS
- Supabase 继续使用外部服务

选择原因：

- 部署和回滚更简单
- 不必侵入当前服务器已有站点结构
- 比裸跑 Node 更稳定、更容易迁移
- 比把反向代理也一起容器化更少变量，适合当前项目规模

## 5. 部署架构

部署链路如下：

`浏览器 -> 服务器现有反向代理 -> Next.js 容器 -> Supabase`

组件职责划分：

- 反向代理负责域名接入、HTTPS、请求转发
- Next.js 容器负责页面渲染、API 路由、管理员会话和 Supabase 服务端访问
- Supabase 负责数据存储与访问控制

容器只暴露应用监听端口，不直接承担证书管理职责。

## 6. 容器化设计

### 6.1 基础镜像

运行时镜像采用 Node.js 20 LTS 的 Debian slim 变体，而不是 Alpine。

原因：

- Next.js 15 及其原生依赖在 glibc 环境中兼容性更稳
- 避免 Alpine + musl 对原生模块带来的额外排查成本
- 当前项目追求的是服务器稳定部署，而不是极限压缩镜像体积

### 6.2 多阶段构建

使用多阶段 `Dockerfile`：

1. 依赖阶段安装生产与构建依赖
2. 构建阶段执行 `npm run build`
3. 运行阶段只保留运行所需产物

这样可以减少最终镜像大小，并隔离构建工具链。

### 6.3 Next.js 构建模式

在 `next.config.ts` 中启用 `output: "standalone"`。

原因：

- Next.js 会生成自包含的服务端运行目录
- 运行阶段不需要完整源码和全部开发依赖
- 更适合 Docker 多阶段构建和生产部署

### 6.4 容器启动方式

运行阶段使用 Next.js standalone 产物启动服务。

启动原则：

- 监听 `0.0.0.0`
- 默认端口 `3000`
- `NODE_ENV=production`
- 禁用 Next telemetry

## 7. Compose 设计

### 7.1 服务数量

`docker-compose.yml` 只定义一个应用服务，例如 `attendance-web`。

### 7.2 端口策略

默认将容器端口映射到服务器本机回环地址：

- `127.0.0.1:3000:3000`

原因：

- 只允许本机反向代理访问
- 避免应用端口直接暴露到公网
- 与“现有代理层统一对外”的架构保持一致

如果服务器没有单独的反向代理，再改为公开绑定端口。

### 7.3 重启策略

使用：

- `restart: unless-stopped`

原因：

- 服务器重启后服务会自动恢复
- 人工停止时不会强制拉起，符合运维预期

### 7.4 健康检查

可选增加健康检查，使用 Node 原生 `fetch` 检查本机 HTTP 响应。

这能帮助 Compose 更快识别应用启动失败，而不必为镜像额外安装 `curl` 或 `wget`。

### 7.5 环境变量加载

Compose 使用单独的生产环境文件，例如：

- `.env.production`

该文件不提交到仓库，服务器上手动创建。

## 8. 环境变量策略

### 8.1 运行所需变量

至少需要：

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_DEFAULT_KEY`
- `SUPABASE_SECRET_KEY` 或 `SUPABASE_SERVICE_ROLE_KEY`
- `ADMIN_PASSWORD`

### 8.2 构建期与运行期

当前项目中 Supabase 相关变量主要由服务端逻辑读取，但变量名包含 `NEXT_PUBLIC_`，因此部署设计需要避免“运行期才注入，构建期缺失”的不确定性。

本次方案采用更稳妥的策略：

- 服务器上构建镜像
- Compose 同时为构建阶段和运行阶段提供同一组公开环境变量

这样可以避免客户端构建结果与运行期配置不一致。

### 8.3 仓库中的变量文件

保留：

- `.env.example` 作为示例模板

新增或约定：

- `.env.production.example` 作为服务器部署模板

不提交：

- `.env.production`

## 9. 反向代理边界

服务器现有反向代理继续保留，不放入 Compose。

代理层需要完成：

- 将域名请求转发到 `127.0.0.1:3000`
- 保留 `Host`、`X-Forwarded-For`、`X-Forwarded-Proto`
- 继续承担 TLS 证书和 HTTPS 跳转

本次仓库改造可以补一份 Nginx 示例配置到文档中，但不强制写入服务器配置目录。

## 10. 需要新增或修改的文件

### 10.1 需要修改

- `next.config.ts`
  - 启用 `output: "standalone"`
- `README.md`
  - 新增 Docker Compose 部署说明
- `.gitignore`
  - 补充部署相关忽略项和 macOS 垃圾文件规则

### 10.2 需要新增

- `Dockerfile`
  - 多阶段构建生产镜像
- `.dockerignore`
  - 排除本地依赖、构建缓存、Git 元数据和 `._*`
- `docker-compose.yml`
  - 定义生产运行服务
- `.env.production.example`
  - 服务器部署变量模板

## 11. macOS 资源叉文件处理

当前仓库内存在大量 `._*` AppleDouble 文件。如果不处理：

- 会污染 Docker 构建上下文
- 可能增加镜像构建体积
- 干扰文件检索与后续版本管理

本次部署改造至少需要在 `.dockerignore` 中排除：

- `._*`
- `.DS_Store`

同时建议在 `.gitignore` 中补上相同规则，避免后续继续进入工作区。

## 12. 部署流程设计

服务器上的标准操作流程为：

1. 拉取项目代码
2. 复制环境变量模板为 `.env.production`
3. 填入 Supabase 与管理员密码配置
4. 执行 `docker compose build`
5. 执行 `docker compose up -d`
6. 由现有反向代理把域名流量转发到 `127.0.0.1:3000`
7. 用浏览器验证首页、学生绑定、签到与后台登录

后续更新流程为：

1. 拉取最新代码
2. 执行 `docker compose build`
3. 执行 `docker compose up -d`

## 13. 错误处理与运维原则

- 应用启动失败时，通过 `docker compose logs -f` 查看日志
- 缺少环境变量时，应在启动或首个请求中明确报错，不允许静默降级
- 端口冲突时，优先检查本机是否已有进程占用 `3000`
- 反向代理异常与应用异常分层排查，避免混在一起定位

## 14. 测试与验收

部署改造完成后，至少验证以下内容：

### 14.1 本地构建验证

- `npm run build` 可以通过
- `docker compose build` 可以通过
- `docker compose up -d` 后首页可访问

### 14.2 业务主流程验证

- 学生绑定流程正常
- 学生签到流程正常
- 后台登录可用
- 后台创建/查看场次正常

### 14.3 部署边界验证

- 服务只监听在反向代理需要的地址和端口
- 修改 `.env.production` 后重建容器可以生效
- 容器重启后服务可恢复

## 15. 风险与权衡

### 15.1 接受的权衡

- 选择 Compose 而不是系统服务，意味着服务器需要安装 Docker
- 选择保留现有反向代理，意味着仓库内不会完全封装整站入口
- 选择在服务器本机构建镜像，意味着公开变量在构建期就需要可用

### 15.2 当前不处理的风险

- 没有引入自动发布流水线，发布仍依赖手工命令
- 没有新增应用专用健康路由时，健康检查粒度较粗
- 没有把反向代理配置托管进仓库时，服务器配置仍需人工维护

## 16. 实施结论

本次实施将围绕一个最小但稳健的生产部署单元展开：

- 一个支持 standalone 输出的 Next.js 应用
- 一个多阶段生产镜像
- 一个单服务 `docker-compose.yml`
- 一套清晰的生产环境变量模板与部署文档

该方案足以把当前项目稳定部署到服务器，同时尽量不引入与项目规模不匹配的复杂度。
